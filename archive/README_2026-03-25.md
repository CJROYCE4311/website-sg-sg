This file is a restart point for the work completed on `2026-03-25`.

## Current State

The canonical model is now:

- `data/scores.csv`
  - stores player round performance
  - `Differential` is the canonical performance metric
  - formula: `(Gross_Score - 70.5) * 113 / 124`
- `data/handicaps.csv`
  - stores handicap snapshots by `Date + Player`
  - fields: `Handicap_Index`, `Course_Handicap`
  - formula: `Handicap_Index * (124 / 113) + (70.5 - 72)`
- `data/financials.csv`
  - stores payout rows by `Date + Player + Category`
  - may stand alone even when score rows later disappear in Squabbit
- `data/player_aliases.json`
  - normalizes name variants before validation and write

## Validation Rules Going Forward

Strict:

- every score row must have a same-date handicap snapshot
- every handicap snapshot must have a same-date score row
- duplicate canonical keys abort ingest

Allowed with audit visibility:

- financial rows may exist without score rows
- this is intentional because payout screenshots may survive even when Squabbit later loses player score data

Best-ball screenshot rule:

- payout information from the best-ball screenshot goes to `financials.csv`
- visible handicap indexes from that same screenshot may be used for `handicaps.csv`
- only create handicap rows from that screenshot for players who also have score rows for that date
- do not create handicap rows from payout-only screenshots when there is no score row

## Recommended Next Steps

1. Decide whether the `Probability` column should remain as-is, be relabeled more explicitly as heuristic, or be replaced with a truer model.
2. When processing future tournaments, do the capture same day whenever possible so:
   - handicap indexes are preserved from the source
   - payouts survive later Squabbit deletions
3. If a best-ball screenshot contains payouts and handicap indexes, split the extraction:
   - payouts to `financials.csv`
   - handicap indexes only for players confirmed in `scores.csv`
