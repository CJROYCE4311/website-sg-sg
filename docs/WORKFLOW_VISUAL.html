Outputs to Verify Before Cleanup

These checks are the safest way to decide what is redundant later. If a file is never produced, never read, or is superseded by a newer generated artifact, it becomes a cleanup candidate.

Reviewed Intermediate

input/tournament_data.json should be the only reviewed handoff into the ingest path.

  • Start from docs/tournament_data.template.json
  • Contains one entry or an update_batch
  • Includes metadata and source notes when needed
  • Keeps handicap snapshots in handicaps[], not scores[]
  • Can be dry-run without canonical writes

Canonical Files

The canonical layer should remain the source of truth for all rebuilds.

  • scores.csv stores round performance and differential
  • handicaps.csv stores handicap snapshots by date and player
  • financials.csv stores payouts by date, player, and category

Generated Files

Everything under website/ should be reproducible from canonical data.

  • HTML reports refresh from scripts/update_site.py
  • index.html gets both the latest results section and the results log links
  • Current recap pages exist for 2025-11-15, 2025-12-20, 2026-01-17, 2026-02-21, and 2026-03-21
  • Methodology bundle refreshes during the same local build
  • No manual edits should be required for generated outputs

Step-by-Step Test Checklist

Run these in order while you validate the pipeline. Once you know which outputs are authoritative at each stage, we can confidently remove redundant scripts, stale generated artifacts, or legacy workflow files.

Test 1

Put a small screenshot batch in input/screenshots/ and confirm the batch boundary is obvious.

Test 2

Start from docs/tournament_data.template.json, then create or review input/tournament_data.json and confirm every needed field exists before running scripts.

Test 3

Run the dry run and verify you get validation feedback without any canonical file mutations.

Test 4

Run the canonical audit and confirm duplicate issues stay at zero and score-to-handicap coverage is complete.

Test 5

Run the real local process and verify the canonical CSVs update only where the payload intended.

Test 6

Open the affected pages in website/ and verify that the same canonical change appears everywhere it should.

Suggested next phase after testing: list which files are canonical, reviewed intermediates, generated outputs, one-off migration utilities, or historical references. That classification will make the cleanup pass much safer.
