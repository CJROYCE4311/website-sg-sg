## Major Workflow Improvements

### 1. Local-first processing model

The active workflow now assumes local processing first and only publishes when explicitly requested.

Key behaviors:

- `scripts/update_site.py` rebuilds locally without committing or pushing by default
- `scripts/process_tournament.py` only commits or pushes when explicit flags are passed
- publish commands stage only `data/` and `website/`
- publish commands refuse to run if unrelated files are dirty

### 2. Canonical data protections

The ingest path was hardened so canonical files are safer to maintain over time.

Implemented protections:

- repeat runs update existing financial rows for the same date/player/category key
- repeat runs update existing handicap rows for the same date/player key
- duplicate canonical keys cause the ingest to abort before write
- score rows require matching handicap snapshots unless an intentional exception is used
- handicap outlier readings are flagged when they are more than 1 standard deviation from a player's historical mean

### 3. Cleaner reviewed payload structure

The reviewed payload model was normalized around:

- `date`
- `metadata`
- `scores[]`
- `financials[]`
- `handicaps[]`

Metadata now standardizes:

- `full_scorecard_available`
- `handicap_list_available`
- `screenshots`
- `source_notes`
- `approximations`

Supporting files:

- `docs/tournament_data.template.json`
- `input/tournament_data.json` as a clean real example during testing

## Current Operator Validation Flow

After each tournament is processed, the intended manual audit is:

1. In Squabbit, open `Leaderboard`, click `Games`, sort `Total Winnings`, and validate `Total Payout` on `website/DataAudit.html`.
2. In Squabbit, open the `Gross` tab, sort by `Total`, and validate `Gross` plus `Total` on `website/DataAudit.html`.
3. In Squabbit, open the gear icon, go to `Players`, sort `HDCP`, then sort `HI` on `website/DataAudit.html` and validate handicap index values.
